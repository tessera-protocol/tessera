\begin{abstract}
\noindent AI agents can now execute shell commands, modify files, send messages, call APIs, and trigger real-world side effects. The main security problem is no longer model output quality alone. It is ambient authority: agents routinely inherit broad, implicit permissions that are hard to scope, hard to revoke, and difficult to audit.

Tessera is a capability-based security layer for agent actions. At the product layer, Tessera provides identity and authority for AI agents. At the technical layer, it issues scoped capabilities, enforces them at runtime, revokes them quickly, and records decision traces for operators and platforms.

Tessera starts with a narrow wedge: runtime-native enforcement inside real agent runtimes such as OpenClaw. The goal is not to solve every aspect of AI safety or semantic policy. The goal is to make sensitive agent actions require explicit, revocable, runtime-enforced authority.

The long-term opportunity is larger. If platforms, runtimes, and operators adopt common authority checks, agents can become easier to trust, easier to constrain, and easier to audit. But that future depends on practical adoption, not protocol elegance. Tessera therefore starts with the most concrete problem first: capability-based security for agent actions.
\end{abstract}

\section{Executive Summary}

AI agents are moving from suggestion to execution. They can already send messages, edit code, access tools, and operate software on behalf of users and organisations. Most of these systems still rely on ambient authority: if an agent has access to a token, tool, or runtime, it can often act with far broader power than was intentionally delegated.

Tessera addresses that problem by introducing scoped, revocable, runtime-enforced authority for agent actions.

The core claim of this paper is simple:

\begin{quote}
agent actions should require explicit capabilities, and those capabilities should be scoped, revocable, enforced at runtime, and auditable.
\end{quote}

This makes Tessera closer to OAuth for agent actions than to a generic identity layer. The point is not merely to name an agent. The point is to control what that agent can do, when it can do it, and how that authority can be stopped.

Today, Tessera's most credible wedge is runtime-native enforcement in real agent runtimes such as OpenClaw. That wedge is intentionally narrow, but it is real: scoped capabilities, runtime hooks, revocation, and audit evidence inside an actual execution path.

\section{The Problem}

\subsection{Agents inherit too much authority}

Modern agent systems are increasingly able to:

\begin{itemize}
  \item execute shell commands,
  \item edit code and files,
  \item send outbound messages,
  \item trigger third-party APIs,
  \item spend money or consume quota,
  \item and coordinate across tools and services.
\end{itemize}

The default security model is often weak. If the runtime can access a tool, the agent can usually attempt to use it. If the operator has already authenticated a service, the agent may inherit those credentials. If there is a manual approval step, it is often either too coarse, too frequent, or too easy to bypass by moving the action elsewhere in the stack.

\subsection{Ambient authority is the real failure mode}

The central failure mode is ambient authority. An agent does not prove explicit authority for each sensitive class of action. It simply acts with whatever authority the surrounding runtime, tool, token, or session already exposes.

That creates predictable problems:

\begin{itemize}
  \item permissions are broader than intended,
  \item delegation is implicit rather than explicit,
  \item revocation is slow or nonexistent,
  \item audit trails are incomplete,
  \item and operators cannot clearly answer why an action was allowed or denied.
\end{itemize}

\subsection{Existing patterns are insufficient}

Most current approaches fall into one of four categories:

\begin{enumerate}
  \item \textbf{Direct tool access.} If the agent has the tool, it can try to use it.
  \item \textbf{Approval spam.} The human is asked repeatedly, which undermines autonomy and encourages approval fatigue.
  \item \textbf{Framework-specific permissions.} Each runtime invents its own policy surface, often with weak portability or poor revocation semantics.
  \item \textbf{Proxy-only controls.} Enforcement happens outside the runtime hot path and can drift from what the agent is actually doing.
\end{enumerate}

None of these gives a clean answer to the core operator questions:

\begin{itemize}
  \item what was this agent allowed to do,
  \item under whose authority,
  \item with what limits,
  \item and how do we stop it now?
\end{itemize}

\section{Thesis}

The thesis of Tessera is that agent actions should require explicit capabilities.

Those capabilities should be:

\begin{itemize}
  \item \textbf{scoped} to specific action classes and limits,
  \item \textbf{runtime-enforced} at the point where the action actually executes,
  \item \textbf{revocable} quickly enough to function as a real kill switch,
  \item and \textbf{auditable} so operators can reconstruct what was requested, allowed, denied, granted, or revoked.
\end{itemize}

This is the deepest technical framing of Tessera:

\begin{quote}
Tessera is a capability-based security layer for agent actions.
\end{quote}

Identity still matters, but it is not the deepest layer. Identity tells a system which principal or agent is involved. Capabilities determine what that agent is allowed to do.

\section{What Tessera Is}

Tessera operates at three levels of framing.

\subsection{Product framing}

Tessera provides identity and authority for AI agents.

\subsection{Technical framing}

Tessera provides capability-based security for agent actions.

\subsection{Operational framing}

Tessera gives agents scoped, revocable, runtime-enforced authority with audit trails.

In practical terms, Tessera is a control layer that sits in or near the runtime path where agent actions execute. It checks whether an agent has a valid capability for the requested action, whether that capability remains in force, and whether the action should be allowed or denied.

The four core pillars are:

\begin{enumerate}
  \item \textbf{Scoped capabilities}
  \item \textbf{Runtime enforcement}
  \item \textbf{Instant revocation}
  \item \textbf{Auditability}
\end{enumerate}

\section{What Tessera Is Not}

Tessera is intentionally narrower than several adjacent narratives.

Tessera is not:

\begin{itemize}
  \item a blockchain system,
  \item a decentralised ledger project,
  \item an antivirus product,
  \item a general semantic moderation engine,
  \item a full AI safety solution,
  \item or a universal ideological identity framework.
\end{itemize}

Tessera does not currently judge whether the content of an action is morally acceptable, manipulative, fraudulent, or otherwise semantically harmful. Its present job is narrower and more concrete: to enforce authority boundaries around action types, scope, expiry, limits, revocation, and operator visibility.

\section{System Model}

The Tessera system model has six core actors and stores.

\subsection{Principal}

The principal is the human or organisation on whose behalf authority is delegated.

\subsection{Agent}

The agent is the runtime subject requesting an action. In practice this may be represented by an agent id, key, session context, or runtime-specific identity.

\subsection{Issuer}

The issuer mints capabilities or credentials that bind delegated authority to an agent within a defined scope.

\subsection{Verifier / guard}

The verifier is the runtime hook or guard layer that evaluates whether a requested action is authorised.

\subsection{Revocation registry}

The revocation registry records whether a capability or delegation instance has been invalidated.

\subsection{Audit log}

The audit log records the decision trace: what was requested, what was presented, what was allowed or denied, when authority changed, and why the result occurred.

In a minimal deployment, these components may all be local. In a larger deployment, the issuer, revocation registry, and audit plane may be managed services.

\section{The Capability Model}

Tessera capabilities define what an agent is allowed to do. A capability may include:

\begin{itemize}
  \item action classes such as \texttt{exec.shell}, \texttt{message.send}, or \texttt{code.write},
  \item quantitative limits such as spend caps or recipient limits,
  \item time bounds such as issued-at and expiry,
  \item runtime or agent binding,
  \item and a unique delegation id for revocation and audit.
\end{itemize}

Illustrative shape:

\begin{lstlisting}
{
  "type": "TesseraCapability",
  "agentId": "main",
  "delegationId": "a4c9b7d8-...",
  "scope": {
    "actions": ["message.send"],
    "maxRecipients": 1
  },
  "issuedAt": 1775220000,
  "expiresAt": 1775223600,
  "issuer": "tessera-demo",
  "signature": "..."
}
\end{lstlisting}

The point of the capability is not to describe everything an agent is. The point is to constrain what it may do.

\section{Enforcement Boundary}

The strongest Tessera integrations sit at real runtime boundaries, not just at dashboards or proxies.

Examples of enforcement boundaries include:

\begin{itemize}
  \item tool invocation hooks such as \texttt{before\_tool\_call},
  \item outbound message hooks such as \texttt{message\_sending},
  \item gateway-side action handlers,
  \item and server-side tool execution boundaries.
\end{itemize}

This matters because enforcement is hardest to bypass when it sits in the execution path itself.

\subsection{Current limitation: semantic enforcement is not solved}

Tessera does not yet perform semantic enforcement.

Today, Tessera can enforce:

\begin{itemize}
  \item whether an action type is permitted,
  \item whether the requested action is in scope,
  \item whether a delegation has expired,
  \item whether a delegation has been revoked,
  \item and whether quantitative bounds are exceeded.
\end{itemize}

It cannot yet reliably determine whether an allowed payload is benign or harmful.

For example:

\begin{itemize}
  \item if an agent has permission to send email or send a message, Tessera can enforce that the send action is permitted within bounds,
  \item but Tessera cannot yet determine whether the content is manipulative, fraudulent, or otherwise unacceptable.
\end{itemize}

This is an intentional honesty boundary. Content-aware policy layers, policy-as-code, and richer contextual evaluation may become future extensions, but they are not current truth.

\section{Runtime-Native Architecture}

Tessera starts as a runtime-native control layer for agent actions.

That means the first credible wedge is not a universal global identity fabric. It is integration into real runtimes where sensitive actions are already happening.

\subsection{Why runtime-native matters}

Runtime-native enforcement has three practical advantages:

\begin{enumerate}
  \item it sits where the action actually executes,
  \item it can fail closed at the boundary,
  \item and it can produce decision evidence tied to the runtime context.
\end{enumerate}

\subsection{OpenClaw as the first wedge}

OpenClaw is a strong example of this wedge because it exposes live runtime hook boundaries. Tessera can attach to those boundaries and make real allow or deny decisions on concrete actions.

Current enforcement is still coarse, but it is real. The present wedge already supports runtime-native control over action classes such as:

\begin{itemize}
  \item \texttt{exec.shell},
  \item \texttt{code.write},
  \item and the runtime hook boundary for \texttt{message.send}.
\end{itemize}

This is important product discipline. Tessera should not claim broad authority coverage where only advisory or proxy-level control exists.

\subsection{Richer runtime context improves policy quality}

Many runtimes today expose only limited action context, often just a tool name and parameters. That is enough to build a credible first wedge, but richer context will improve:

\begin{itemize}
  \item audit quality,
  \item approval flows,
  \item contextual policy decisions,
  \item and operator confidence.
\end{itemize}

\section{Revocation}

Revocation is one of Tessera's strongest practical advantages.

If an operator decides an agent should stop acting, that authority should not linger until the next login rotation, human approval flow, or deployment cycle. It should stop immediately enough to function as a true kill switch.

This is why Tessera treats revocation as part of the live authority model rather than a background administrative concern.

In practice, Tessera revocation should support:

\begin{itemize}
  \item explicit invalidation of a delegation instance,
  \item one-to-one revocation identity,
  \item fast verifier checks,
  \item and clear operator visibility that authority has been shut down.
\end{itemize}

The implementation may vary by deployment, but the product requirement is fixed:

\begin{quote}
revocation must be fast enough to shut down agent authority in practice, not just in theory.
\end{quote}

\section{Auditability and Decision Trace}

Enforcement alone is not enough. Operators also need to understand what happened.

Tessera therefore treats auditability as a first-class feature, not a secondary log sink.

A useful decision trace should answer:

\begin{itemize}
  \item what action was requested,
  \item what capability or credential was presented,
  \item what runtime or agent requested it,
  \item whether the action was allowed or denied,
  \item why that decision occurred,
  \item and when authority was granted, revoked, or expired.
\end{itemize}

This audit timeline is valuable both operationally and commercially. It makes agent authority legible to developers, operators, enterprise reviewers, and eventually platforms that need to classify agents as verified, restricted, or revoked.

\section{Policy Model and Role Templates}

One of the main adoption bottlenecks is policy definition.

Most developers and operators do not want to author raw capability JSON from scratch. Even when a low-level policy model is expressive, it is not the best interface for everyday adoption.

The practical product direction is:

\begin{quote}
policies should feel like choosing a role, not authoring a schema.
\end{quote}

\subsection{Role templates}

Tessera should increasingly expose role templates and preset capability profiles such as:

\begin{itemize}
  \item \textbf{Researcher / read-only}
  \item \textbf{Developer / local dev}
  \item \textbf{Assistant / communications}
  \item \textbf{Purchaser / bounded spend}
  \item \textbf{Custom / advanced}
\end{itemize}

Raw policies should remain available, but role-first onboarding is the adoption path most users will actually tolerate.

\subsection{Safe defaults}

Templates are not just convenience. They create safer defaults, clearer expectations, and faster operator understanding. They also make policy review and audit easier because the authority shape is easier to recognise.

\section{Deployment Model}

The deployment strategy should be managed first, federated later.

\subsection{Hosted Tessera Cloud}

The near-term default should be a hosted Tessera control plane. This is the fastest path to adoption, the cleanest operator experience, and the easiest way to ship:

\begin{itemize}
  \item issuer services,
  \item revocation checks,
  \item audit timelines,
  \item policy management,
  \item and role-template onboarding.
\end{itemize}

\subsection{Self-hosted enterprise Tessera}

Enterprises will eventually need self-hosted deployment for policy control, internal trust boundaries, compliance needs, and data locality.

\subsection{Federated ecosystem deployment}

Longer term, Tessera should support multiple issuers, multiple revocation registries, and interoperable verification across organisational boundaries.

That federation does not require a blockchain. It can be built with standard issuer trust, registry distribution, and verifier interoperability.

The important sequencing is adoption first, federation later.

\section{Go-to-Market and Product Wedge}

Tessera should not lead with universal claims. It should lead with the strongest real wedge.

That wedge is:

\begin{quote}
runtime-native authority control for agent actions in real runtimes.
\end{quote}

The starting value proposition is concrete:

\begin{itemize}
  \item scoped delegation,
  \item runtime enforcement,
  \item fast revocation,
  \item and audit trails for sensitive agent actions.
\end{itemize}

This is a more believable adoption story than claiming to already be global infrastructure for the agentic web.

\subsection{The adoption analogy}

In adoption terms, the long-term ambition is closer to HTTPS than to a niche developer add-on: a world where non-use becomes visibly unsafe or unacceptable, and platforms begin to label agents as verified, restricted, or revoked.

In user experience terms, the eventual product could feel closer to a VPN:

\begin{itemize}
  \item one toggle,
  \item green means protected,
  \item and most of the machinery remains invisible.
\end{itemize}

But the core technical analogy remains capability delegation. Tessera is closer to OAuth for agent actions.

\section{Long-Term Vision}

The long-term vision is that Tessera becomes a common authority layer for agent actions.

That does not mean claiming infrastructure status in the present tense. It means pursuing a path where:

\begin{itemize}
  \item runtimes expose real enforcement hooks,
  \item operators expect scoped and revocable authority,
  \item platforms distinguish between verified and unverified agents,
  \item and non-use of authority controls becomes visibly risky.
\end{itemize}

If that happens, Tessera or a system like it could become part of the standard security posture of agent systems.

But that outcome depends on adoption, not ideology. The gating factor is whether Tessera solves concrete operator problems better than ambient authority, repeated prompts, or framework-specific controls.

\section{Conclusion}

The core security problem in agent systems is ambient authority.

Agents should not operate with broad, implicit, unrevocable permissions. Sensitive actions should require explicit capabilities that are scoped, enforced at runtime, revocable, and auditable.

That is the central claim of Tessera.

Today, Tessera starts with a narrow but credible wedge: runtime-native enforcement in real agent runtimes. Over time, it can expand into a broader authority layer for agents, but only if it remains grounded in real execution boundaries, honest product limits, and practical operator workflows.

The right quality bar is not grandeur. It is whether a technically literate reader can say:

\begin{quote}
this is narrower than I expected, but much more real.
\end{quote}
